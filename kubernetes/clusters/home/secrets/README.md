# Kubernetes secrets

Cluster secrets are committed encrypted with SOPS and the age backend, and Flux decrypts them in-cluster as it reconciles. The encrypted files are safe to keep in a public repository; only the cluster holds the private age key, stored as the `sops-age` secret in `flux-system`.

`/.sops.yaml` at the repo root defines the rule: any file matching `kubernetes/.*/secrets/.*\.sops\.yaml` has its `data` and `stringData` fields encrypted to the cluster's age recipient. The `cluster-secrets` Kustomization decrypts them at apply time.

## Layout

Secrets are grouped by purpose, each folder carrying its own kustomization:

- `bootstrap/`: cluster provisioning and node bootstrap, covering the Cluster API providers, the burst cluster kubeconfig and token, the Hetzner CCM/CSI token, and the image-pull credential.
- `platform/`: networking edge, covering Cloudflare DNS-01 and tunnel credentials and the Tailscale auth key.
- `identity/`: Authentik and its database bootstrap.
- `apps/`: per-application credentials for the workloads.

The file name is the consumer; the produced Kubernetes Secret name lives inside each manifest's `metadata.name`.

The Hetzner token in `bootstrap/hcloud.sops.yaml` is the same account token used by the separate vigil project, so any rotation must be coordinated across both repos.

## Bootstrap secrets that stay out of GitOps

`cattle-system/bootstrap-secret` is generated by the Rancher chart on first install and consumed during initial admin setup; committing it would fight the controller and serve no purpose once the admin password is set.

## Adding or editing a secret

Edit in place; SOPS re-encrypts on save:

```
sops kubernetes/clusters/home/secrets/<folder>/<name>.sops.yaml
```

For a new secret, write the plaintext manifest, encrypt it (`sops --encrypt --in-place <name>.yaml` then rename to `<name>.sops.yaml`), and list it in that folder's `kustomization.yaml` so Flux applies it.

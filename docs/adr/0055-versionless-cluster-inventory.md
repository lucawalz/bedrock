# 0055. Track the cluster inventory as a versionless topology map

## Context

`docs/inventory.md` is generated by `scripts/gen-inventory.sh` from the home cluster's HelmReleases, and a CI job regenerates it and fails on any drift. The table carried four columns: component, namespace, chart, and chart version. The version came straight from each HelmRelease's `spec.chart.spec.version`.

That version column duplicated state that already has a single home. Under the Flux GitOps model from [0002](0002-nixos-flakes-flux-gitops.md) the HelmRelease manifest is the source of truth for which chart version the cluster runs, and Renovate edits that field directly. The generated table cached the same number, so every Renovate version bump left the committed inventory one revision behind and turned the drift check red. Renovate treats a red check as a red branch and stops auto-merging, so routine dependency PRs stalled and each one needed a manual regenerate commit to clear.

The hosted Renovate app cannot run `gen-inventory.sh` as a post-upgrade task, and `main` is a protected branch. Nothing could regenerate the file inside the bump PR without granting CI write access back into the repository, which runs against the outward-only flow of GitOps.

## Decision

Drop the version column. The inventory now lists component, namespace, and chart only: a stable topology map of what runs where, not a snapshot of pinned revisions.

The chart version stays solely in the HelmRelease manifest, recoverable from Git like any other declared state and reconciled to the cluster by Flux. A disaster recovery restores from the same manifests, so the table never needed its own copy of the number.

With the volatile field gone, the generated file changes only when a workload is added or removed, which is always a human change. The drift check returns to a strict `git diff --exit-code` and is promoted to a required status check on the `protect-main` ruleset, so a topology change cannot merge without its regenerated table.

## Options considered

- Write the regenerated table back into the bump PR from CI. Rejected. It needs a long-lived token or a GitHub App with write access so the pushed commit re-triggers the required checks, which adds a credential to rotate and points CI back into the repository against the GitOps direction.
- Regenerate on merge and commit to `main`. Rejected. The `protect-main` ruleset forbids direct pushes, and routing the bot through its own pull request carries the same write-credential cost.
- Keep the version but make the drift check non-blocking. Rejected. It leaves a knowingly stale version in a disaster recovery document, which is worse than not recording it.
- Have Renovate bump the version inside the table with a custom manager. Rejected. It would hardcode a chart-to-registry mapping that already lives in the HelmRepository definitions, trading one duplication for another.

## Consequences

Renovate version bumps no longer touch the inventory, so dependency PRs stay green and auto-merge without manual regeneration. The drift check is now meaningful: it fires only on real topology changes and blocks them until the table is regenerated. The table no longer answers which chart version is deployed; that question is answered by the HelmRelease in Git, which was always its authoritative source.

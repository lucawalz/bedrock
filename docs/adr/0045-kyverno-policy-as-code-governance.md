# 0045. Govern workloads with Kyverno policy-as-code in audit-first rollout

## Context

The triage in [0043](0043-triage-production-readiness-findings.md) codified a hardening posture by hand: it set litellm resource limits, switched homepage and litellm to read-only root filesystems, and wrote down which trade-offs were deliberate. That posture lived only as prose and as the discipline of whoever last edited a manifest. Nothing in the cluster checked that the next workload kept to it. The first-party manifests already follow a consistent shape — every container carries CPU and memory requests and limits, runs as non-root, sets `allowPrivilegeEscalation: false`, drops all capabilities, and pins a fixed image tag — but that shape was convention, not a contract. A scanner sweep catches drift after the fact and off-cluster; it does not stop a non-compliant Pod from admitting, and it does not surface a running cluster's compliance state in one place. The cluster needed the conventions turned into machine-checked policy, enforced at admission where the manifests are clean and audited everywhere else, with a single source of policy truth shared between the cluster and CI.

## Decision

Adopt Kyverno as the policy engine, run every policy audit-first, and graduate to enforce only where offline evidence proves first-party manifests already pass. Kyverno installs as a Flux HelmRelease in a dedicated `kyverno` namespace, under `infrastructure/security/`, with background scanning and policy reports enabled so the cluster's compliance state is continuously evaluated, not only checked at admission. Five `ClusterPolicy` resources codify the existing conventions: resource requests and limits on every container, `runAsNonRoot`, `allowPrivilegeEscalation: false`, `capabilities.drop: [ALL]`, and a fixed image tag that is never `:latest`. They are adapted from Kyverno's pod-security and best-practices library rather than hand-authored from scratch.

Every policy ships `validationFailureAction: Audit`, so admitting the roughly eighty existing workloads — most of them upstream Helm charts that do not meet the restricted baseline — never breaks. Graduation to enforce is decided per namespace from `validationFailureActionOverrides`, which lets one policy stay audit cluster-wide while enforcing in named namespaces. The `blog` and `homepage` namespaces are enforced, because their only workloads are first-party Deployments that pass all five policies. Every other namespace stays audit.

Policy Reporter installs as a second Flux HelmRelease in the same namespace, with its UI, Prometheus metrics exporter, and Kyverno plugin enabled. Its ServiceMonitor carries the `release: kube-prometheus-stack` label so the existing Prometheus scrapes PolicyReport metrics, and its Grafana dashboards render into the `monitoring` namespace with the `grafana_dashboard` sidecar label so they load into the existing Grafana with no new instance. The UI is exposed at `policy-reporter.syslabs.dev` behind the same Traefik IngressRoute and Authentik forward-auth middleware that front the Longhorn and Velero dashboards under [0038](0038-authentik-sso-for-internal-dashboards.md).

The same `ClusterPolicy` resources are the single source of policy truth for CI. A `kyverno-policy` job in the manifest-validation workflow runs `kyverno test` against authored unit tests, where a compliant resource must pass and a violating one must fail, and then runs `kyverno apply` of the policies against the rendered first-party manifests, failing the build on any violation. The policies are authored once and consumed by both admission and CI.

## Options considered

- Kyverno, chosen. Policies are Kubernetes-native YAML, so the same `ClusterPolicy` resource drives admission control, background audit, and the CI gate with no second language. It emits native `PolicyReport` resources that Policy Reporter renders into a UI and Prometheus metrics, it scans existing workloads in the background rather than only at admission, and `validationFailureActionOverrides` expresses the audit-first-then-enforce rollout on a single object.
- OPA Gatekeeper. Rejected. Policies are written in Rego, a separate language to learn and test, which splits the source of truth between the cluster and any shift-left check and adds a translation step the single-operator cluster does not need. Its reporting and ecosystem around dashboards and metrics is thinner than Kyverno's PolicyReport-plus-Policy-Reporter path.
- Keep the scanner sweep from [0043](0043-triage-production-readiness-findings.md) and add no admission control. Rejected. A periodic off-cluster scan catches drift late and cannot stop a non-compliant Pod from admitting, and it leaves the conventions as prose rather than a checked contract.

## Consequences

The conventions are now a contract. A first-party workload in `blog` or `homepage` that drops a resource limit, runs as root, or floats to `:latest` is rejected at admission, and the same violation fails CI before it merges, because both read the one policy set. Everywhere else the policies audit only, so the existing upstream charts keep admitting and their gaps surface as PolicyReports in the Policy Reporter UI and as Prometheus metrics in the existing Grafana, turning the [0043](0043-triage-production-readiness-findings.md) accepted-trade-off list into a live, queryable view rather than a static note. The audit-first default means upstream Helm workloads that cannot meet the restricted baseline stay accepted-and-audited, consistent with the triage posture, until each is either remediated or its namespace is graduated.

The graduation decision was made offline, because Flux reconciles from the main branch and the live PolicyReports for this change do not exist until it merges. `kyverno apply` of the five policies against the three authored first-party manifests — the blog and homepage Deployments and the litellm database-init Job — is the evidence. The blog and homepage Deployments passed all five policies on the first run, which is why both namespaces graduated. The litellm init Job failed four of them: it carried no resource requests or limits and no security context, a clear oversight in a first-party manifest, so it was corrected to non-root with dropped capabilities, no privilege escalation, and bounded resources, after which all three manifests pass. Its namespace, `postgres`, stays audit regardless, because it also hosts the upstream PostgreSQL chart that does not meet the baseline. The cost is two more controllers to run and keep current, and an enforce surface that must be widened deliberately, one namespace at a time, each time backed by the same offline `kyverno apply` evidence before a namespace is added to the overrides.

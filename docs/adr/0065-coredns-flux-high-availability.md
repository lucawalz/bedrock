# 0065. Adopt CoreDNS into Flux for DNS high availability

## Context

In-cluster DNS ran as the k3s bundled CoreDNS addon, owned by a k3s.cattle.io Addon object. It shipped as a single replica with no PodDisruptionBudget, so the loss of the node holding that pod took cluster DNS down until it rescheduled, and every service lookup stalled in the meantime. The addon ownership also meant any manual edit to the CoreDNS Deployment, Service, or ConfigMap was reverted the next time k3s reconciled its addons, so the resource could not be tuned in place. That left DNS as the one critical-path service the topology work in [0053](0053-ha-critical-path-survives-node-loss.md) deliberately left single-replica.

## Decision

Disable the k3s CoreDNS addon with `--disable=coredns`, matching the existing convention that already disables servicelb, traefik, and local-storage on the control-plane node, and manage CoreDNS as raw manifests in Flux under the infrastructure controllers. The Flux copy runs two replicas with a required pod anti-affinity on `kubernetes.io/hostname` so the two pods always land on different nodes, and a PodDisruptionBudget with `minAvailable: 1` so a voluntary disruption never drains both at once. This follows the GitOps model from [0001](0001-k3s-on-nixos.md) and [0002](0002-nixos-flakes-flux-gitops.md), and reuses the same required-anti-affinity-plus-budget pattern the ingress path adopted in [0053](0053-ha-critical-path-survives-node-loss.md).

The Service keeps the cluster DNS address pinned at 10.43.0.10, the value every pod inherits through its resolv.conf, and the ConfigMap carries the CoreDNS Corefile verbatim from the addon alongside a static NodeHosts block for the three nodes.

## Options considered

- CoreDNS as Flux-managed raw manifests at two replicas, chosen. Durable, versioned, and highly available, with spread and a disruption budget under the cluster's own control.
- Scaling the k3s addon in place with an HPA or a manifest override. Rejected, the same way it was declined in [0053](0053-ha-critical-path-survives-node-loss.md): the addon reverts edits on reconcile, so any override fights k3s rather than owning the resource.
- Leaving CoreDNS single-replica. Rejected, because it remained the last single point of failure on the critical path after the ingress and database work.

## Consequences

Cluster DNS now survives the loss of any single node, spread across two nodes with a disruption budget, and its configuration lives in git rather than in a reconciled addon. Two constraints follow. The Service ClusterIP must stay pinned at 10.43.0.10, because changing it would strand every pod that already resolved DNS to the old address. The NodeHosts block is now static in git and no longer regenerated by k3s, so a change to a node IP requires a manual ConfigMap update.

Cutover carries a sharper risk than a brief gap, through a bootstrap deadlock. Flux fetches its manifests from a remote git host, and that fetch depends on cluster DNS. Once the addon deletes CoreDNS, source-controller can no longer resolve the git host, so Flux cannot fetch the CoreDNS manifests and cannot recreate the service it just removed, and DNS does not self-heal. The recovery, and the standing procedure whenever CoreDNS is fully down, is to apply the manifests directly from the working tree with `kubectl apply -k kubernetes/infrastructure/controllers/coredns/`, which reaches the API server without DNS and does not wait on Flux. Once the pods are ready, source-controller resolves the git host again and Flux resumes ownership on its next reconcile. The initial cutover hit exactly this deadlock, so the durable sequence is to disable the addon and then recreate CoreDNS with that direct apply rather than trusting Flux to recreate it.
